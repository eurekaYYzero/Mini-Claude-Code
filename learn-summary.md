# 学习记录
## 第一章
### Loop
完整的多轮：发送消息给模型->模型回复->调用工具->拿到执行结果(Tool_Result)->回写工具结果到state(messages)

## 第二章
### 工具
生产级工具：
- 每个 `tool_use` 块必须有对应的 `tool_result`
  - 通过 `tool_use_id` 进行关联
- `user` / `assistant` 消息必须严格交替
  - 不能连续出现两条相同角色的消息
- 只能使用协议中定义的字段
  - 传入内部元数据会导致 `400` 错误

以上三点是最基础的要求，随着业务复杂还需要引入
权限、上下文、MCP Client、文件系统、query跟踪等机制

当多个工具同时存在时，需要考虑分发的问题，即有一个工具列表传输给llm，然后根据返回的内容来决定调用哪个
## 第三章
### plan模式
大模型在执行长任务经常会有遗忘和幻觉，因此需要维护一个session级别的todolist
list里面每一个任务都维护一些变量：执行状态，多少轮没执行了，执行结果
每个任务执行过后，让模型来更新todolist
从这个任务开始，小模型就不太行了，经常会出现长任务中间就结束的情况
## 第四章
### 子智能体
父智能体随着轮次增加，上下文会很长，因此需要子智能体
一个最简单的子智能体就是从空白上下文开始，或者从父智能体的上下文开始，独立维护自己的messages
完成工作后只返回执行的结果
子智能体只能执行有限的工具，有轮次限制、错误处理，让子流程可以正常退出
子agent的代码和主agent几乎一样
对于现有逻辑，增加一个task的工具来生成子agent

这一章开始，执行更加不稳定了，其实应该引入REACT机制，让模型校验和反思，工作是否完成

## 第五章
### Skill
Skill其实主要解决的还是Prmopt过长的问题，让提示和资源能够渐进式引入
实际上就是维护一个skill目录，新增一个load_skill的工具

这是某类任务才需要的做法或知识：skill
这是需要长期记住的事实或偏好：memory
这是更稳定的全局规则：CLAUDE.md

在system里面加入name和desc，load_skill之前在本地加载好所有的skill，然后按照模型返回来加载

## 第六章
### 上下文
三步分层处理，每一次循环自动缩减tool_result，只保留最近的，剩下的直接说调用过
如果超出token，把当前的保存到文件中，然后让llm总结，最后只保留总结内容和文件路径

## 第七章
### 权限
调用工具前需要进行权限校验：
1、危险命令需要拒绝
2、分plan、act场景执行不同的工具
3、有一些需要用户二次确认的
4、最后才是直接可执行的

## 第八章
### Hook
不是所有的逻辑都要放到主循环，需要有hooks机制
1、在工具调用前后，调用hooks
2、使用约定好的hooks文件来实现要执行什么，最好是名称和函数文件的映射

## 第九章
### Memory
记忆是无法从当前环境得知的内容才保存为长期记忆
1、用户明确纠正的case和确认执行成功的case
2、当次对话的信息或者任务应当用task来保存
3、全局级别的规范应该放到.md
4、不容易从对话或环境推理出来的做为保存，下次使用的时候可以再校验一下

记忆对agent也是一个工具，包括写入记忆，查询所有记忆，注入到系统提示词这几个主要功能，同时记忆还需要有索引文件加快处理
